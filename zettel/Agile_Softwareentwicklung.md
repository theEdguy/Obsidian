# [[Agile_Softwareentwicklung]]

- **Kernkonzept:** Die [[Agile_Softwareentwicklung]] ist ein iteratives und inkrementelles [[Vorgehensmodell]] im [[Software_Engineering]], das auf Flexibilität, kontinuierliche Anpassung und enge Zusammenarbeit zwischen [[Entwicklungsteam|Entwicklungsteams]] und [[Stakeholder|Stakeholdern]] setzt, um schnell auf Veränderungen zu reagieren und funktionsfähige [[Software]] in kurzen Zyklen (z. B. [[Sprints]]) zu liefern. Sie basiert auf dem [[Spiralmodell]] von Boehm (1988) und betont die frühzeitige Identifikation und Bewältigung von [[Risiko|Risiken]] durch regelmäßige [[Feedbackschleife|Feedbackschleifen]] und [[Retrospektive|Retrospektiven]] im [[Scrum]]- oder [[Kanban]]-Rahmenwerk.
- **Nutzen & Zweck:** Die [[Agile_Softwareentwicklung]] existiert, um die starren und oft langwierigen traditionellen [[Vorgehensmodell|Vorgehensmodelle]] wie das [[Wasserfallmodell]] zu überwinden, die häufig zu unflexiblen Lösungen, veralteten [[Anforderung|Anforderungen]] und mangelnder [[Kundenzufriedenheit]] führen. Durch iterative [[Entwicklungsprozess|Entwicklungsprozesse]] ermöglicht sie frühzeitiges Feedback, regelmäßige Lieferungen von Teilprodukten und adaptive Planung, was die Qualität, Transparenz und Effizienz von [[Softwareprojekt|Softwareprojekten]] deutlich verbessert. Zu den zentralen Vorteilen zählen:
- **Frühzeitige Lieferung funktionsfähiger [[Software]]**: Durch iterative [[Entwicklung]] können Teilprodukte bereits frühzeitig eingesetzt und getestet werden, was die [[Time-to-Market]] verkürzt.
- **[[Kundenzentrierung]]**: Regelmäßige [[Feedbackschleife|Feedbackschleifen]] mit dem [[Kunde|Kunden]] (z. B. durch [[User_Story|User Stories]] oder das [[Product_Backlog]]) stellen sicher, dass das Endprodukt den tatsächlichen [[Anforderung|Anforderungen]] entspricht.
- **[[Risikomanagement]]**: Durch kontinuierliche Evaluation und Anpassung (z. B. in [[Retrospektive|Retrospektiven]] oder [[Daily_Standup|Daily Standups]]) werden [[Risiko|Risiken]] früh erkannt und behoben, was die [[Projektstabilität]] erhöht.
- **Transparenz und Zusammenarbeit**: Agile [[Methode|Methoden]] fördern die Selbstorganisation von [[Entwicklungsteam|Teams]] (z. B. durch [[Scrum]]-Rollen wie [[Scrum_Master]] oder [[Product_Owner]]) und eine offene Kommunikation, was die [[Teamproduktivität]] steigert.
- **Anpassungsfähigkeit**: Änderungen in den [[Anforderung|Anforderungen]] oder Rahmenbedingungen können flexibel berücksichtigt werden, ohne den gesamten [[Projektplan]] zu gefährden, was besonders in dynamischen Umgebungen mit unklaren oder sich ändernden [[Zielvorgabe|Zielvorgaben]] entscheidend ist.
- **Abgrenzung & Grenzen:** Die [[Agile_Softwareentwicklung]] grenzt sich von starren, sequenziellen [[Vorgehensmodell|Modellen]] wie dem [[Wasserfallmodell]] ab, die eine vollständige Planung zu Projektbeginn voraussetzen und weniger Flexibilität für Änderungen bieten. Dennoch hat auch Agilität Grenzen und typische Herausforderungen:
- **Eignung für Projekte**: Projekte mit klar definierten, stabilen [[Anforderung|Anforderungen]] (z. B. sicherheitskritische Systeme wie [[Avionik]] oder medizinische Geräte) profitieren möglicherweise weniger von Agilität, da hier eine detaillierte Vorabplanung und Dokumentation erforderlich ist.
- **Dokumentation**: Agile [[Methode|Methoden]] legen weniger Wert auf umfangreiche Vorabdokumentation, was zu [[Wissensverlust]] führen kann, wenn [[Teammitglied|Teammitglieder]] wechseln oder das [[Projektwissen]] nicht ausreichend transferiert wird. Hier können [[Wiki]]-Systeme oder [[Knowledge_Management]]-Tools unterstützen.
- **Kundenbeteiligung**: Agile [[Prozess|Prozesse]] erfordern eine aktive und kontinuierliche Einbindung des [[Kunde|Kunden]], was in der Praxis oft schwer umsetzbar ist, insbesondere wenn dieser nicht verfügbar oder unentschlossen ist. Dies kann zu Verzögerungen oder Fehlentwicklungen führen.
- **Skalierung**: Agile [[Rahmenwerk|Rahmenwerke]] wie [[Scrum]] oder [[Kanban]] sind primär für kleine bis mittelgroße [[Entwicklungsteam|Teams]] optimiert. Bei großen [[Softwareprojekt|Projekten]] mit vielen [[Team|Teams]] (z. B. in [[SAFe]] oder [[LeSS]]) steigt die Komplexität der Koordination, was zusätzliche [[Skalierungsframework|Skalierungsframeworks]] oder [[Koordinationsmechanismus|Koordinationsmechanismen]] erfordert.
- **Kulturelle Hürden**: Agile Prinzipien setzen eine offene [[Fehlerkultur]], [[Selbstorganisation]] und flache Hierarchien voraus, was in traditionell hierarchischen oder bürokratischen Organisationen auf Widerstand stoßen kann. Die Einführung agiler [[Methode|Methoden]] erfordert daher oft einen kulturellen Wandel.
- **Messbarkeit und Kontrolle**: Traditionelle [[Metrik|Metriken]] wie Meilensteine oder detaillierte [[Projektplanung|Projektpläne]] sind in agilen [[Projekt|Projekten]] schwerer anwendbar, was die Fortschrittskontrolle und [[Stakeholder]]-Kommunikation erschweren kann. Hier kommen alternative [[Metrik|Metriken]] wie [[Velocity]] oder [[Burndown_Chart|Burndown-Charts]] zum Einsatz.
- **Vertragliche Rahmenbedingungen**: Agile [[Projekt|Projekte]] sind oft schwer mit klassischen Festpreisverträgen vereinbar, da diese eine feste [[Anforderung|Anforderungsspezifikation]] und Budgetierung voraussetzen. Hier können [[Time-and-Material-Verträge]] oder [[Agile_Vertragsmodell|agile Vertragsmodelle]] eine Lösung bieten.

### Java-Beispiel: Vereinfachtes Scrum-Board
```java
import java.util.ArrayList;
import java.util.List;

class ScrumBoard {
    private List<String> todo;
    private List<String> inProgress;
    private List<String> done;
    private String sprintGoal;
    
    public ScrumBoard(String sprintGoal) {
        this.todo = new ArrayList<>();
        this.inProgress = new ArrayList<>();
        this.done = new ArrayList<>();
        this.sprintGoal = sprintGoal;
    }
    
    public void addTask(String task) {
        todo.add(task);
    }
    
    public void moveToInProgress(String task) {
        if (todo.remove(task)) {
            inProgress.add(task);
        }
    }
    
    public void completeTask(String task) {
        if (inProgress.remove(task)) {
            done.add(task);
        }
    }
    
    public void printBoard() {
        System.out.println("Sprint Goal: " + sprintGoal);
        System.out.println("To Do: " + todo);
        System.out.println("In Progress: " + inProgress);
        System.out.println("Done: " + done);
    }
    
    public int getVelocity() {
        return done.size();
    }
}
```

### Beispiel für eine [[User_Story]] mit Akzeptanzkriterien und [[Definition_of_Done]]:
```plaintext
**User Story:**
Als [[Endbenutzer]] möchte ich meine [[Task|Tasks]] per Drag & Drop priorisieren können, damit ich meine Arbeit effizienter organisieren kann.

**Akzeptanzkriterien:**
- [[Task|Tasks]] können per Drag & Drop in der [[Benutzeroberfläche]] sortiert werden.
- Prioritäten werden in der [[Datenbank]] persistiert und sind für alle [[Teammitglied|Teammitglieder]] sichtbar.
- Änderungen werden in Echtzeit synchronisiert (z. B. via [[WebSocket]]).
- Die Priorisierung funktioniert auf Desktop- und Mobilgeräten ([[Responsive_Design]]).

**Definition of Done (DoD):**
- Code ist gemäß [[Code_Review]]-Richtlinien geprüft und freigegeben.
- [[Unit_Test|Unit-Tests]] und [[Integrationstest|Integrationstests]] sind implementiert und erfolgreich.
- Dokumentation (z. B. [[API]]-Dokumentation, [[Benutzerhandbuch]]) ist aktualisiert.
- Feature ist auf der [[Staging_Umgebung]] deployed und vom [[Product_Owner]] abgenommen.
```
