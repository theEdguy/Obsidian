# [[Scrum]]

- **Kernkonzept:** [[Scrum]] ist ein [[agiles_Framework|agiles Rahmenwerk]] für [[Projektmanagement]], [[Softwareentwicklung]] und die Wartung komplexer [[Produkt|Produkte]], das auf kurzen [[Iteration|Iterationen]] ([[Sprint|Sprints]]), [[Selbstorganisation]] von [[cross-funktionale_Teams|cross-funktionalen Teams]] und kontinuierlicher [[Verbesserung]] durch regelmäßige [[Reflexion]] und [[Feedback]]-Schleifen basiert. Es definiert klare [[Rolle|Rollen]] wie [[Product_Owner]], [[Scrum_Master]] und [[Entwicklungsteam]] sowie zentrale [[Artefakt|Artefakte]] wie [[Produkt-Backlog|Product Backlog]] und [[Sprint-Backlog]], um [[Transparenz]], [[Inspektion]] und [[Anpassung]] zu fördern und schnelle Reaktion auf [[Änderung|Änderungen]] zu ermöglichen.
- **Nutzen & Zweck:** [[Scrum]] existiert, um die Herausforderungen traditioneller, starrer [[Projektmanagement]]-Methoden wie dem [[Wasserfallmodell]] zu überwinden, die oft inflexibel auf [[Änderung|Änderungen]] reagieren, unklare [[Anforderung|Anforderungen]] haben und wenig Raum für kontinuierliche [[Verbesserung]] lassen. Es löst konkrete [[Problem|Probleme]] wie lange [[Entwicklungszyklus|Entwicklungszyklen]], mangelnde [[Kundeneinbindung]] und ineffiziente [[Kommunikation]] in [[Team|Teams]], indem es kurze, fokussierte [[Iteration|Iterationen]] ([[Sprint|Sprints]]) nutzt, regelmäßiges [[Feedback]] einholt und [[cross-funktionale_Teams|cross-funktionale Teams]] fördert. Dadurch wird die [[Produktivität]] gesteigert, [[Risiko|Risiken]] frühzeitig erkannt und die [[Anpassungsfähigkeit]] an sich ändernde [[Markt|Märkte]] oder [[Kundenbedürfnis|Kundenbedürfnisse]] verbessert. Häufige Lieferungen funktionsfähiger [[Inkrement|Inkremente]] steigern die [[Kundenzufriedenheit]], verkürzen die [[Time-to-Market]] und maximieren den [[Wert]] für [[Stakeholder|Stakeholder]]. Durch transparente [[Artefakt|Artefakte]] (z. B. [[Produkt-Backlog|Product Backlog]]) und [[Ereignis|Ereignisse]] (z. B. [[Daily_Scrum]], [[Sprint_Review]]) wird die [[Zusammenarbeit]] zwischen [[Entwicklungsteam|Entwicklungsteams]] und [[Stakeholder|Stakeholdern]] effizienter. Regelmäßige [[Meeting|Meetings]] wie die [[Retrospektive]] unterstützen die kontinuierliche [[Verbesserung]] des [[Prozess|Prozesses]] und des [[Produkt|Produkts]], wodurch die [[Produktqualität]] erhöht wird. Zudem etabliert [[Scrum]] [[Transparenz]], [[Inspektion]] und [[Anpassung]] als zentrale [[Prinzip|Prinzipien]], um die [[Effizienz]] und [[Flexibilität]] in [[Projekt|Projekten]] zu erhöhen und die Lieferung hochwertiger [[Produkt|Produkte]] in kürzeren Zyklen sicherzustellen. Als [[Prozessmodell]] für die [[Softwareentwicklung]] ermöglicht es komplexe [[Projekt|Projekte]] effizienter und transparenter zu gestalten, indem es kurze [[Feedback]]-Schleifen, kontinuierliche [[Anpassung]] und enge [[Kollaboration]] zwischen [[Entwicklungsteam|Entwicklern]] und [[Stakeholder|Stakeholdern]] fördert. Dies reduziert [[Risiko|Risiken]] und verbessert die [[Produktqualität]] durch iterative [[Planung]] und frühes [[Feedback]].
- **Abgrenzung & Grenzen:** [[Scrum]] ist nicht geeignet für [[Projekt|Projekte]] mit stabilen, klar definierten [[Anforderung|Anforderungen]] oder hoher [[Dokumentationspflicht]], da der [[Overhead]] der agilen [[Prozess|Prozesse]] dann unnötig ist. Im Vergleich zu klassischen [[Vorgehensmodell|Vorgehensmodellen]] wie dem [[Wasserfallmodell]] fehlt eine detaillierte [[Vorabplanung]], was bei sicherheitskritischen oder stark regulierten [[Projekt|Projekten]] problematisch sein kann. [[Scrum]] ist kein Ersatz für [[technische_Praktik|technische Praktiken]] wie [[Test-Driven_Development|Test-Driven Development (TDD)]], [[Continuous_Integration]] oder [[Pair_Programming]], die in [[Extreme_Programming|Extreme Programming (XP)]] stärker betont werden. Alternativen wie [[Kanban]] eignen sich besser für kontinuierliche [[Arbeitsfluss|Arbeitsflüsse]] ohne feste [[Iteration|Iterationen]], während [[Scrum]] durch seine feste [[Sprint-Dauer]] charakterisiert ist. Hybride Ansätze wie [[Scrumban]] kombinieren Elemente von [[Scrum]] und [[Kanban]], um die Vorteile beider [[Framework|Frameworks]] zu nutzen. [[Scrum]] erfordert eine hohe Disziplin und [[Selbstorganisation]] des [[Entwicklungsteam|Entwicklungsteams]] sowie eine aktive Beteiligung der [[Stakeholder]]. Zu ambitionierte [[Sprint|Sprints]] können zu [[Überlastung]] führen, während eine unklare [[Priorisierung]] der [[Anforderung|Anforderungen]] im [[Produkt-Backlog|Product Backlog]] die [[Effizienz]] des [[Team|Teams]] beeinträchtigen kann. Im Gegensatz zu [[Kanban]] fehlt [[Scrum]] eine explizite Begrenzung der parallelen [[Aufgabe|Aufgaben]] ([[Work_in_Progress|Work in Progress]]), und neue [[Anforderung|Anforderungen]] können während eines [[Sprint|Sprints]] nicht ohne Weiteres aufgenommen werden. [[Scrum]] sollte nicht genutzt werden, wenn [[Projekt|Projekte]] klar definierte, unveränderliche [[Anforderung|Anforderungen]] haben und eine lineare, vorhersehbare Entwicklung erfordern, wie z. B. in stark regulierten [[Umgebung|Umgebungen]] mit festen Vorgaben oder bei kleinen, einfachen [[Projekt|Projekten]], bei denen der Overhead der [[Rolle|Rollen]] und [[Ereignis|Events]] unnötig ist. Zudem ist [[Scrum]] weniger geeignet für [[Team|Teams]], die keine [[Selbstorganisation]] oder kontinuierliche [[Verbesserung]] anstreben. Hierarchische oder unerfahrene [[Team|Teams]] können mit der geforderten [[Disziplin]] und [[Autonomie]] überfordert sein. Als [[Prozessmodell]] unterscheidet es sich von klassischen [[Vorgehensmodell|Modellen]] durch seine Flexibilität und iterative Natur, ist jedoch weniger geeignet für [[Projekt|Projekte]] mit strikten [[Compliance]]- oder [[Dokumentationsanforderung|Dokumentationsanforderungen]].

public class UserStory {
    private String id;          // Eindeutige Kennung (z. B. "US-12")

    private String acceptanceCriteria; // Akzeptanzkriterien für die Umsetzung

        this.id = id;
        this.beschreibung = beschreibung;
        this.storyPoints = storyPoints;
        this.prioritaet = prioritaet;
        this.status = Status.TO_DO;
        this.acceptanceCriteria = acceptanceCriteria;
    }

    public void aktualisiereStatus(Status neuerStatus) {
        this.status = neuerStatus;
    }

// Beispiel für eine einfache Scrum-Sprint-Planung (Pseudocode)
import java.util.ArrayList;
import java.util.Arrays;
import java.util.List;

public class SprintPlanning {
    public static void main(String[] args) {
        List<String> productBacklog = Arrays.asList(
            "[[Benutzerregistrierung]] implementieren",
            "[[Passwort-Reset]]-Funktion hinzufügen",
            "[[Datenbankoptimierung]]",
            "[[UI]] für [[Dashboard]] überarbeiten"
        );
        
        List<String> sprintBacklog = new ArrayList<>();
        int sprintCapacity = 3; // Anzahl der [[Aufgabe|Aufgaben]], die im Sprint umsetzbar sind
        
        // Priorisierte [[Aufgabe|Aufgaben]] aus dem Product Backlog in den Sprint Backlog übernehmen
        for (int i = 0; i < sprintCapacity; i++) {
            sprintBacklog.add(productBacklog.get(i));
        }
        
        System.out.println("Sprint Backlog für den nächsten Sprint:");
        sprintBacklog.forEach(System.out::println);
    }
}

// Beispiel für eine einfache Scrum-ähnliche Sprint-Planung in Java
class SprintExample {
    private String goal;
    private int durationDays;
    private List<String> backlogItems;

    public SprintExample(String goal, int durationDays) {
        this.goal = goal;
        this.durationDays = durationDays;
        this.backlogItems = new ArrayList<>();
    }

    public void addBacklogItem(String item) {
        backlogItems.add(item);
    }

    public void start() {
        System.out.println("Sprint gestartet: " + goal +
                           " (Dauer: " + durationDays + " Tage)");
        System.out.println("Backlog-Items:");
        backlogItems.forEach(System.out::println);
    }
}

public class ScrumExample {
    public static void main(String[] args) {
        SprintExample sprint = new SprintExample("Mitgliederverwaltung implementieren", 14);
        sprint.addBacklogItem("Datenbank für Mitglieder anlegen");
        sprint.addBacklogItem("GUI für Mitgliederverwaltung erstellen");
        sprint.addBacklogItem("Authentifizierung integrieren");
        sprint.start();
    }
}

// Beispiel für ein einfaches Scrum-Board (Task-Verwaltung in Java)
import java.util.ArrayList;
import java.util.List;

public class ScrumBoard {
    private List<Task> todo;
    private List<Task> inProgress;
    private List<Task> done;
    private List<UserStory> backlog;
    private List<UserStory> sprintBacklog;

    public ScrumBoard() {
        this.todo = new ArrayList<>();
        this.inProgress = new ArrayList<>();
        this.done = new ArrayList<>();
        this.backlog = new ArrayList<>();
        this.sprintBacklog = new ArrayList<>();
    }

    public void addTask(Task task) {
        todo.add(task);
    }

    public void moveToInProgress(Task task) {
        if (todo.remove(task)) {
            inProgress.add(task);
        }
    }

    public void moveToDone(Task task) {
        if (inProgress.remove(task)) {
            done.add(task);
        }
    }

    public void moveToSprintBacklog(UserStory story) {
        backlog.remove(story);
        sprintBacklog.add(story);
    }

    public void startWork(UserStory story) {
        sprintBacklog.remove(story);
        inProgress.add(new Task(story.getDescription(), ""));
    }

    public void complete(UserStory story) {
        Task task = new Task(story.getDescription(), "");
        if (inProgress.remove(task)) {
            done.add(task);
        }
    }
}

class Task {
    private String description;
    private String assignee;

    public Task(String description, String assignee) {
        this.description = description;
        this.assignee = assignee;
    }
    
    @Override
    public boolean equals(Object obj) {
        if (this == obj) return true;
        if (obj == null || getClass() != obj.getClass()) return false;
        Task task = (Task) obj;
        return description.equals(task.description);
    }
    
    @Override
    public int hashCode() {
        return description.hashCode();
    }
}
