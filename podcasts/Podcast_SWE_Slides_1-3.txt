## **Definitionen**

### **1. Software Engineering**
**Definition**: The application of engineering principles to the design, development, testing, deployment, and maintenance of software in a systematic, repeatable, and measurable manner.
**Key Aspects**:
- **Systematic**: Follows structured processes (e.g., [[Software Development Life Cycle (SDLC)]]).
- **Disciplined**: Adheres to standards (e.g., [[ISO/IEC 12207]] for software life cycle processes).
- **Quantifiable**: Metrics (e.g., [[cyclomatic complexity]], [[defect density]]) measure quality.
**Why It Matters**: Without engineering rigor, software becomes fragile, unmaintainable, and prone to failures (e.g., [[Therac-25]] disaster due to unchecked software errors).

### **2. Prof. Dr. Thomas Fuchß**
**Role**: The course instructor, likely an expert in [[software architecture]], [[agile methodologies]], or [[formal methods]]. His academic and industry background ensures alignment with current research and practical challenges (e.g., [[DevOps]], [[cloud-native development]]).

---

## **Beispiele / Code**

### **Example: The Cost of Ignoring Software Engineering**
**Scenario**: A startup builds a mobile app without engineering principles.
1. **Lack of Requirements Engineering**:
   - No clear [[user stories]] or [[non-functional requirements]] (e.g., scalability for 1M users).
   - **Result**: App crashes under load; users abandon it.
2. **Poor Architecture**:
   - Monolithic design with no [[separation of concerns]].
   - **Result**: A single bug in the payment module crashes the entire app.
3. **No Testing**:
   - No [[unit tests]] or [[integration tests]].
   - **Result**: A critical bug reaches production, causing financial losses.
4. **Technical Debt**:
   - Quick fixes pile up without [[refactoring]].
   - **Result**: Future updates take months; competitors outpace the startup.

**Code Snippet (Anti-Pattern)**:
```python
# No separation of concerns; business logic mixed with UI
def process_order(order):
    if order["items"] > 10:  # Hardcoded threshold
        print("Discount applied")
    else:
        print("No discount")
    # Database connection directly in the function
    db.connect()
    db.save(order)
```
**Why It’s Bad**:
- **Violates [[Single Responsibility Principle (SRP)]]**: The function handles discounts, UI, and database logic.
- **No testability**: Impossible to mock `db.connect()` for unit tests.
- **Brittle**: Changing the discount threshold requires modifying the function.

**Refactored Version**:
```python
class DiscountCalculator:
    def calculate(self, order) -> float:
        return 0.1 if order["items"] > 10 else 0.0

class OrderProcessor:
    def __init__(self, discount_calculator: DiscountCalculator, db: Database):
        self.discount_calculator = discount_calculator
        self.db = db

    def process(self, order):
        discount = self.discount_calculator.calculate(order)
        self.db.save(order)
```
**Improvements**:
- [[Dependency Injection]]: `DiscountCalculator` and `Database` are injected.
- [[Modularity]]: Each class has a single responsibility.
- **Testable**: `DiscountCalculator` can be unit-tested in isolation.

# **Lecture: Development Processes, Object-Oriented Principles, and System Architecture**
*Based on Folie 2 of 387 – Prof. Dr. Th. Fuchß, Hochschule Karlsruhe*

---

## **TL;DR**
This slide introduces the **fundamental pillars of software engineering**: structured [[development process]]s, [[object-oriented programming]] (OOP), [[requirements engineering]], [[system architecture]], and [[design patterns]]. It emphasizes the **evolution from ad-hoc development to systematic methodologies**, highlights the **unified perspective of OOP**, and outlines key techniques like [[Use Cases]], [[MVC]], and [[design patterns]]. The slide also signals the transition from **analysis to architecture**, setting the stage for deeper exploration in subsequent lectures.

---

## **1. Definitions: Core Concepts on the Slide**

### **A. Development Processes: Yesterday vs. Today**
- **Historical Context**:
  Early software development (1960s–1970s) was chaotic—projects often failed due to **lack of structure**, leading to the **"software crisis"**. The **waterfall model** (1970) introduced sequential phases (requirements → design → implementation → testing), but its rigidity caused issues in dynamic projects.
  - **Why it matters**: Without a process, teams risk **inconsistent deliverables**, **missed deadlines**, and **technical debt**.
- **Modern Approaches**:
  Agile methodologies (e.g., [[Scrum]], [[Kanban]]) prioritize **iterative development**, **customer collaboration**, and **adaptability**. Hybrid models (e.g., [[V-Modell XT]]) blend structure with flexibility.

### **B. Object-Oriented Programming (OOP)**
OOP is a **paradigm** that models software as **interacting objects**, each encapsulating **data (attributes)** and **behavior (methods)**. Key principles:
1. **Encapsulation**: Hiding internal state (e.g., `private` variables in Java) to prevent misuse.
2. **Inheritance**: Reusing code via hierarchical relationships (e.g., `Dog extends Animal`).
3. **Polymorphism**: Objects of different classes responding to the same method call (e.g., `shape.draw()` for `Circle` or `Square`).
4. **Abstraction**: Focusing on **what** an object does, not **how** (e.g., abstract classes in Java).

### **C. Inheritance vs. Composition/Aggregation/Association**
| Concept          | Definition                                                                 | Example                                                                 |
|------------------|----------------------------------------------------------------------------|-------------------------------------------------------------------------|
| **Inheritance**  | "Is-a" relationship (e.g., `Car` **is a** `Vehicle`).                      | `class ElectricCar extends Car { ... }`                                |
| **Composition**  | Strong "part-of" relationship (child’s lifecycle tied to parent).         | `Engine` cannot exist without `Car` (`Car` has-a `Engine`).            |
| **Aggregation**  | Weak "has-a" relationship (child can exist independently).                | `Department` contains `Employee` (but `Employee` can switch departments). |
| **Association**  | Generic relationship (no ownership implied).                              | `Student` enrolls in `Course` (bidirectional link).                    |

- **Why prefer composition over inheritance?**
  Inheritance creates **tight coupling**; composition promotes **modularity** and **reusability** (favored in [[SOLID]] principles).

### **D. Requirements & Use Cases**
- **Requirements**: Formal specifications of what a system must do (functional vs. non-functional).
  - **Example**: "The system shall authenticate users via OAuth2."
- **Use Cases**: Narrative descriptions of **actor-system interactions** to capture **functional requirements**.
  - **Structure**:
    1. **Primary Actor**: User initiating the use case.
    2. **Preconditions**: System state before execution.
    3. **Main Success Scenario**: Step-by-step success path.
    4. **Extensions**: Alternative flows (e.g., "User enters wrong password").
  - **Why it matters**: Use cases bridge the gap between **vague stakeholder needs** and **technical design**.

### **E. System Architecture: MVC & Observer Pattern**
- **MVC (Model-View-Controller)**:
  - **Model**: Business logic/data (e.g., `User` class).
  - **View**: UI representation (e.g., HTML template).
  - **Controller**: Mediates input (e.g., `UserController` handling HTTP requests).
  - **Use Case**: Separates concerns to enable **scalability** and **testability**.
- **Observer Pattern**:
  - **Definition**: A **publisher-subscriber** mechanism where objects (**observers**) react to changes in a **subject**.
  - **Example**: A `WeatherStation` (subject) notifies `Display` (observer) of temperature updates.
  - **Why it matters**: Decouples components, enabling **real-time updates** (e.g., event-driven systems).

### **F. Design Patterns**
- **Creational Patterns**: Control object creation (e.g., [[Singleton]], [[Factory Method]]).
  - **Example**: `DatabaseConnection` as a Singleton to avoid redundant connections.
- **Structural Patterns**: Simplify relationships between objects (e.g., [[Adapter]], [[Bridge]]).
  - **Adapter Pattern**: Converts incompatible interfaces (e.g., legacy `PaymentProcessor` to new `StripeGateway`).
- **Behavioral Patterns**: Define communication between objects (e.g., [[Observer]], [[Strategy]]).

---

## **2. Example: From Requirements to Architecture**
**Scenario**: Building a **Library Management System**.

### **Step 1: Requirements & Use Cases**
- **Requirement**: "Librarians shall add/remove books."
- **Use Case**:
  - **Actor**: Librarian.
  - **Main Flow**:
    1. Librarian selects "Add Book."
    2. System prompts for title/author.
    3. System validates input and saves to database.
  - **Extensions**:
    - Duplicate ISBN → "Book already exists."
    - Invalid input → "Title must be >3 characters."

### **Step 2: OOP Design**
```java
// Model (Encapsulation)
class Book {
    private String isbn;
    private String title;
    // Getters/setters omitted for brevity
}

// Controller (MVC)
class BookController {
    public void addBook(String isbn, String title) {
        Book book = new Book(isbn, title);
        Database.save(book); // Composition: Book depends on Database
    }
}

// View (UI)
class BookView {
    public void displayBook(Book book) {
        System.out.println("Title: " + book.getTitle());
    }
}
```

### **Step 3: Architecture (MVC + Observer)**
- **Observer**: Notify librarians when a book is overdue.
  ```java
  class OverdueNotifier implements Observer {
      @Override
      public void update(Observable o, Object arg) {
          System.out.println("ALERT: Book " + arg + " is overdue!");
      }
  }
  ```

### **Step 4: Design Patterns**
- **Factory Method**: Create different `Book` types (e.g., `EBook`, `PhysicalBook`).
- **Adapter**: Integrate a third-party `PaymentService` with the library’s `BillingSystem`.

This slide lays the **foundation for disciplined software engineering**:
1. **Processes** ensure **repeatability** and **quality**.
2. **OOP** provides a **unified, modular** approach to modeling real-world systems.
3. **Requirements & Use Cases** translate **business needs** into **actionable design**.
4. **Architecture (MVC, Observer)** and **patterns** solve **recurring problems** with proven solutions.

**Critical Takeaway**:
Ignoring these principles leads to **spaghetti code**, **technical debt**, and **failed projects**. Conversely, applying them systematically results in **scalable**, **maintainable**, and **adaptable** systems.

# **Lecture Monologue: Foundations of Software Engineering and Agile Development**

---

## **TL;DR**
This slide outlines the **core literature** underpinning the lecture, spanning **object-oriented analysis and design (OOAD)**, **iterative development**, **agile methodologies (Scrum)**, and **modern software engineering principles**. The selected texts provide a **comprehensive framework** for understanding how software systems are designed, developed, and delivered in practice. Ignoring these principles risks **technical debt**, **poor scalability**, and **failed project outcomes**.

---

## **Definitions: Key Concepts and Their Significance**

### **1. Object-Oriented Analysis and Design (OOAD)**
**Definition**: A software development approach that models systems using **objects** (instances of classes) to represent real-world entities, their attributes, and behaviors. OOAD emphasizes **encapsulation**, **inheritance**, and **polymorphism** to create modular, reusable, and maintainable code.

**Why It Matters**:
- **Modularity**: Breaks complex systems into manageable components.
- **Reusability**: Reduces redundant code (e.g., inheritance hierarchies).
- **Maintainability**: Changes in one part of the system have minimal ripple effects.

**Consequence of Ignoring OOAD**:
- **Spaghetti Code**: Unstructured logic leads to **unmaintainable** systems.
- **High Coupling**: Components become overly dependent, making updates risky.
- **Example**: A monolithic `UserManager` class handling authentication, logging, and database operations violates **Single Responsibility Principle (SRP)**.

---

### **2. Iterative Development**
**Definition**: A software development lifecycle (SDLC) where requirements and solutions evolve through **repetitive cycles** (iterations). Each iteration delivers a **potentially shippable product increment**.

**Key Characteristics**:
- **Feedback Loops**: Continuous refinement based on stakeholder input.
- **Risk Mitigation**: Early detection of flaws (e.g., design flaws in Iteration 1).
- **Adaptability**: Adjusts to changing requirements (unlike rigid waterfall models).

**Consequence of Ignoring Iterative Development**:
- **Late Discovery of Flaws**: Bugs or misaligned features surface only in later stages, increasing costs.
- **Example**: A banking app developed in one go might fail user acceptance testing (UAT) due to overlooked edge cases (e.g., invalid transaction inputs).

---

### **3. Scrum (Agile Framework)**
**Definition**: A **lightweight agile framework** for managing complex projects by dividing work into **sprints** (time-boxed iterations, typically 2–4 weeks). Scrum roles include:
- **Product Owner**: Defines priorities (e.g., backlog items).
- **Scrum Master**: Facilitates the process, removes impediments.
- **Development Team**: Delivers increments.

**Core Artifacts**:
- **Product Backlog**: Prioritized list of features/bugs.
- **Sprint Backlog**: Tasks selected for the current sprint.
- **Increment**: Potentially releasable product version.

**Why Scrum?**
- **Transparency**: Daily stand-ups and burndown charts track progress.
- **Flexibility**: Adapts to changing business needs.
- **Consequence of Ignoring Scrum**:
  - **Chaotic Development**: Lack of structure leads to missed deadlines.
  - **Example**: A team without sprint planning may overcommit, causing burnout and poor-quality deliverables.

---
### **4. Modern Software Engineering (Sommerville, 2020)**
**Definition**: A holistic approach integrating **engineering principles** (e.g., requirements engineering, testing, DevOps) with **software development**.

**Key Areas**:
- **Requirements Engineering**: Eliciting and validating stakeholder needs.
- **Testing**: Unit, integration, and system testing to ensure reliability.
- **DevOps**: Bridging development and operations for continuous delivery.

**Consequence of Poor Engineering**:
- **Security Vulnerabilities**: Unvalidated inputs lead to exploits (e.g., SQL injection).
- **Example**: A mobile app without automated testing may crash on edge-case inputs (e.g., low battery).

---

## **Example: Applying OOAD and Scrum in Practice**
**Scenario**: Developing an **e-commerce checkout system**.

### **OOAD Implementation**
1. **Classes**:
   - `User` (attributes: `id`, `email`; methods: `login()`).
   - `Cart` (methods: `addItem()`, `calculateTotal()`).
   - `PaymentProcessor` (methods: `processCreditCard()`).
2. **Design Pattern**: Use the **Strategy Pattern** for payment methods (e.g., credit card vs. PayPal).
   ```java
   interface PaymentStrategy {
       void pay(int amount);
   }
   class CreditCardPayment implements PaymentStrategy { ... }
   ```
3. **Benefit**: If the client adds a new payment method (e.g., cryptocurrency), only the `PaymentStrategy` hierarchy changes—**no modification to `Cart` or `User`**.

### **Scrum Workflow**
1. **Sprint 1**:
   - **Backlog Item**: "Implement user authentication."
   - **Sprint Goal**: Secure login with OAuth.
   - **Daily Stand-up**: "Yesterday, I integrated Google OAuth; today, I’ll test edge cases."
2. **Sprint 2**:
   - **Backlog Item**: "Add shopping cart functionality."
   - **Increment**: Users can now add/remove items.
3. **Outcome**: After 3 sprints, the system is **incrementally tested, reviewed, and deployed**.

**Failure Scenario**: Without OOAD, the system might use a single `Checkout` class handling everything—**violating SRP** and making future changes painful. Without Scrum, the team might work in silos, leading to **integration hell**.

---

## **Fazit: The Core Takeaway**
This slide underscores that **successful software development** relies on:
1. **OOAD** for **scalable, maintainable** systems.
2. **Iterative Development** for **adaptability** and **early feedback**.
3. **Scrum** for **structured collaboration** and **risk reduction**.
4. **Modern Engineering Practices** to ensure **reliability** and **security**.

**Final Warning**: Skipping these principles doesn’t just lead to **technical debt**—it risks **project failure**. As Fred Brooks noted in *The Mythical Man-Month*, **"Adding manpower to a late software project makes it later."** The literature here provides the **tools to avoid that fate**.
