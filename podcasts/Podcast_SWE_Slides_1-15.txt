## **1. Introduction to Software Engineering**
**TL;DR**
Software Engineering is the systematic, engineering-driven approach to developing, maintaining, and evolving large-scale software systems using proven methods, tools, and principles.

**Definitions**
- **Software Engineering**: The application of engineering principles to the design, development, testing, and maintenance of software systems.
- **Systematic Development**: Structured, repeatable processes ensuring reliability, scalability, and maintainability.
- **Object-Oriented (OO)**: A paradigm organizing software as objects (data + behavior), enabling modularity and reusability.

**Examples / Explanations**
- **Traditional vs. Modern Processes**:
  - *Yesterday*: Ad-hoc, unstructured development (e.g., "code-and-fix" approach).
  - *Today*: Iterative models like [[Scrum]] or [[Unified Process]], integrating OO principles.
- **OO Concepts**:
  - **Inheritance**: A subclass (`Car`) inherits properties from a superclass (`Vehicle`).
  - **Composition**: A `Car` *has* an `Engine` (strong lifecycle dependency).
  - **Aggregation**: A `Department` *contains* `Employees` (weak dependency).

**Fazit**
Software Engineering transforms chaotic coding into a disciplined, scalable discipline by combining engineering rigor with OO design.

## **2. Core Challenges in Software Engineering**
**TL;DR**
Modern software engineering faces decomposition, reuse, abstraction, and the integration of AI—requiring structured methods to manage complexity.

**Definitions**
- **Decomposition**: Breaking systems into manageable modules (e.g., [[MVC]] pattern).
- **Redundancy**: Avoiding duplicate code via [[Design Patterns]] (e.g., [[Singleton]]).
- **Abstraction**: Hiding complexity (e.g., [[UML]] diagrams).
- **AI in SE**: Automating routine tasks (e.g., code generation via [[LLMs]]).

**Examples / Explanations**
- **Dependencies**:
  - *Explicit*: Direct method calls (`order.calculateTotal()`).
  - *Implicit*: Hidden side effects (e.g., global state).
- **AI Impact (2025)**:
  - KI suggests next steps (e.g., "Implement a `PaymentService`").
  - Risks: Over-reliance on AI-generated code without validation.

**Fazit**
Balancing automation with human oversight is critical to prevent technical debt and ensure correctness.

## **3. Engineering vs. Ad-Hoc Development**
**TL;DR**
Engineering methods (quantitative/qualitative) ensure reproducibility, while ad-hoc approaches lead to unpredictable outcomes.

**Definitions**
- **Quantitative Methods**: Measuring metrics (e.g., defect rates, performance).
- **Qualitative Methods**: Understanding *why* (e.g., developer interviews).
- **Empirical Research**: Experiments, surveys, and case studies to validate techniques.

**Examples / Explanations**
- **Controlled Experiments**:
  - *Scenario*: Compare two sorting algorithms (e.g., [[QuickSort]] vs. [[MergeSort]]).
  - *Metric*: Execution time under varying input sizes.
- **Case Studies**:
  - Analyzing a failed project to identify process gaps (e.g., lack of [[Requirements Engineering]]).

**Fazit**
Data-driven decisions reduce guesswork and improve software quality.

## **4. Key Objectives in Software Engineering**
**TL;DR**
Productivity, quality, cost reduction, and automation are achieved through standardized processes, tools, and frameworks.

**Definitions**
- **Productivity**: Output per unit of effort (e.g., lines of code/hour).
- **Quality**: Reliability, maintainability, and correctness (e.g., [[Test-Driven Development]]).
- **Automation**: Tools like [[CI/CD]] pipelines to streamline deployment.

**Examples / Explanations**
- **Frameworks**:
  - [[Spring Framework]] (Java) for modular applications.
  - [[React]] (JavaScript) for UI components.
- **Processes**:
  - [[Agile]] (iterative) vs. [[Waterfall]] (sequential).

**Fazit**
Standardization enables scalability and reduces errors across teams.
